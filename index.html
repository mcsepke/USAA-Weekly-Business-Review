Feature Requests (AHA)
Open product ideas submitted by USAA to Rubrik engineering
Separate filters for Oracle standalone DBs and Data Guard groups
RBK-I-6967
New

Problem: RSC Object Protection Report shows "Oracle DB" as the only filter, but SLA can be assigned at the Data Guard Group level, causing misleading "No SLA assigned" entries for DG members.

Business Impact: Requires manual reconciliation after every report, reducing confidence in protection status.

Linked Case: 1232081 · Product Area: Platform / RSC Reporting

Email reports to use RSC local time zone instead of UTC
RBK-I-6581
New

Problem: Emailed reports default to UTC, creating a discrepancy with local time displayed in the RSC UI.

Proposed Solution: Update the email reporting engine to inherit and apply the customer's local time zone settings.

Linked Case: 1172031 · Product Area: Platform / RSC Reporting

Configurable, more frequent replication reporting schedule in RSC
RBK-I-6861
New

Problem: Replication status and lag are only refreshed once per day, creating a visibility gap during incidents and audits.

Proposed Solution: Allow configurable report frequency (e.g., every 1/3/6 hours or cron-style), plus a lightweight real-time replication status view.

Linked Case: 1165501 · Product Area: Platform / Replication

Configurable replication lag alerts and notifications
RBK-I-6862
New

Problem: No granular alerting for replication lag. Lag can grow silently, creating RPO/RTO risk that's only discovered during incidents.

Proposed Solution: Configurable lag thresholds (warning/critical) at the SLA Domain or replication relationship level, with email, webhook, syslog/SIEM, and in-product notification channels.

Linked Case: 1165501 · Product Area: Platform / Replication

RSC UI support for managing NFS export IP/subnet overrides (nfsExportsOverride)
RBK-I-6903
High Impact

Problem: Tied to Qualys QID-66002 finding and USAA host-list scalability risk. Currently requires API-only management, creating security exception overhead.

Business Impact: Pre-requisite to expansion with Physical Clusters, Edge, Virtual, and Cloud Clusters. Opportunities linked to this item — Identity Resilience and Universal Cloud Licenses in AWS — represent over $1M in ACV combined.

Linked Case: 1089707 · Product Area: Platform / Security

This week's action items are responses to NAS Cloud Direct questions submitted by USAA via email ahead of this session. Answers are provided below for review and discussion.
Q1 — How often are updates applied to NCD?
NCD Operations

NAS Cloud Direct is delivered as a SaaS service. The control plane and NCD console are updated by Rubrik on a roughly weekly cadence, with additional hotfixes rolled out as needed. Sites on the standard auto-upgrade channel receive these updates automatically — no customer maintenance windows or patching steps are required.

The stateless VM image itself is periodically refreshed (new OVA/VHD builds), but redeployment is only needed when onboarding a new site or when Rubrik Support specifically recommends it (e.g., for a major update). Existing VMs continue working against the updated service automatically.

Q2 — Do we get any notification on updates?
NCD Operations

There is no per-release software update email for NCD — updates are applied transparently on the SaaS side and existing jobs, policies, and data continue to run as-is. For operational awareness, two notification mechanisms are available:

Job / failure notifications: NCD can send email alerts on job failures. SMTP settings are configured via a Rubrik Support request.

RSC notifications: In Rubrik Security Cloud, email notification preferences can be configured for events (backup failures, anomalies, etc.) for any recipients you choose.

In summary: you are notified about operational impact (failed jobs, anomalies), but not every background software release.

Q3 — If an NCD VM is corrupted and has to be rebuilt, what happens?
NCD Resilience

NCD VMs are stateless. All file metadata, indexes, policies, and job history live in Rubrik's managed control plane — not on the VM disk. If a VM is lost or corrupted, the recovery steps are:

1. Download a fresh NCD VM image for that same site.
2. Deploy it on your hypervisor (vSphere, Hyper-V, AHV, etc.) with network access to the NAS systems, backup targets, and the NCD/RSC endpoints.
3. Power it on and wait until the console shows "Network connection healthy" and "Services connected."

Once connected, the VM automatically re-attaches to the existing site in the control plane and recovers:

— All previously added systems and shares/exports
— Existing backup/archive policies and targets
— Historical backups and indexes

Backups and restores resume incrementally from where they left off. No re-seeding of data is required. Rubrik protects control-plane metadata with multi-AZ durability in its own AWS accounts.

Q4 — How does it work when adding multiple NCD VMs?
NCD Scaling

NAS Cloud Direct scales out by adding more stateless VMs to the same site — not by creating separate, independent appliances. Each VM can drive up to ~5 Gbps, and task concurrency scales with the number of VMs deployed.

In Rubrik Security Cloud, you see one NAS Cloud Direct site/cluster per environment (e.g., "SAT-PROD"). The individual VMs are not exposed as separate objects in RSC — they are internal data movers associated with that site. Managing five NCD VMs for SAT-PROD means they all appear collectively as a single NCD entry in RSC's inventory and reports (with "NCD Site" as a column), rather than five separate clusters.

Q5 — What are the steps to "move NCD to RSC"?
NCD Roadmap

There are two distinct operating modes today:

Standalone NCD (legacy): Policies and management live in the NCD console.
NCD-via-RSC: NCD is fully integrated with RSC for SLA Domains, RBAC, reporting, retention lock, and more — while the data path stays VM → target.

Current state: Full migration of existing standalone NCD sites into RSC-managed mode is not yet GA. The migration tooling for existing customers is targeted for FY27. Until that migration wizard is available, a full lift-and-shift of an existing NCD site into SLA-Domain-managed RSC mode is not customer-performable.

What's available today for existing NCD customers without a full migration:

— Events & webhooks for NCD
— RBAC for restore-only and view-only access
— Multi-file (shopping-cart) restore from RSC
— NAS CD reporting (Protection / Object Capacity / Compliance reports)

Recommendation: For new NCD sites, deploy in NCD-via-RSC mode from day one. For USAA's existing standalone NCD sites, enable the partial RSC integration pieces today, and once Rubrik releases the migration wizard (FY27) we'll have a guided, in-place path to move to full RSC management — without re-seeding data.
